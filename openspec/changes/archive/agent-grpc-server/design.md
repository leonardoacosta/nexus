## Context

nexus-agent is the per-machine daemon in the Nexus peer-to-peer architecture. It needs to serve
session data to TUI clients over gRPC. The `notify` crate provides cross-platform file watching
(inotify on Linux, FSEvents on macOS). Proto types are generated by spec 1 (proto-and-codegen)
in nexus-core.

## Goals / Non-Goals

- Goals: Working gRPC server with session RPCs, file-based session discovery, in-memory registry
- Non-Goals: Health endpoint (spec 5), session streaming/WebSocket (spec 5), TUI client (spec 4),
  session hooks (future), authentication (future)

## Decisions

### Session Registry: `Arc<SessionRegistry>` with internal `RwLock`

- Decision: `SessionRegistry` owns a `tokio::sync::RwLock<HashMap<String, Session>>` internally.
  Shared via `Arc<SessionRegistry>` between the gRPC service and file watcher.
- Alternatives: `DashMap` (heavier dep, not needed at this scale), `Mutex` (blocks readers
  unnecessarily), channel-based actor (over-engineered for ~100 sessions max).

### File Watcher: `notify` with debounce

- Decision: Use `notify::RecommendedWatcher` with a 100ms debounce on file events. Parse the
  full `sessions.json` on each change event rather than diffing.
- Rationale: sessions.json is small (<10KB typically). Full parse is simpler and avoids stale
  state bugs. The debounce prevents redundant parses from rapid writes.

### Session Type Tracking

- Decision: All sessions discovered via file watching are marked `SessionType::AdHoc`. Future
  specs will add hook-registered sessions with richer metadata.
- Rationale: File watching cannot determine how a session was started. Ad-hoc is the safe default.

### Proto-to-Serde Conversion

- Decision: Implement `From<nexus_core::session::Session>` for the proto `Session` message type
  in `grpc.rs`. Keep both type systems until spec 3/4 migrates the TUI to proto-native types.
- Rationale: Gradual migration. The serde types are the source of truth for now; proto is the
  wire format.

### Port Binding

- Decision: Bind gRPC server on `0.0.0.0:7400`. Tailscale handles access control at the network
  layer — no application-level auth needed for MVP.
- Rationale: Matches PRD. Tailscale ACLs restrict which machines can reach port 7400.

## Risks / Trade-offs

- File watcher may miss events if the file is written atomically (rename) vs in-place — mitigated
  by watching the parent directory and filtering for the target filename.
- No graceful degradation if sessions.json doesn't exist yet — mitigated by logging a warning
  and continuing with an empty registry until the file appears.
- Proto type drift: if spec 1 changes proto definitions, conversion code here must update.
  Mitigated by compile-time type checking (trait impls break on field changes).
