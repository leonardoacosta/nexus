# Product Requirements Document — Nexus Production Hardening

> Generated: 2026-03-23
> Source artifacts: scope-lock.md, MVP PRD (archived)
> Phase: production-hardening (post-MVP)

## 1. Vision & Problem Statement

**Vision:** Make Nexus reliable enough to run unattended across homelab + Mac, complete the M3
TUI feature set, and establish a test foundation that prevents regressions.

**Problem:** The MVP works but requires manual intervention. The agent has port binding issues
after restart, excessive memory usage (450M+), no graceful shutdown (clients see connection
reset), and no auto-reconnect. Three planned M3 features remain unshipped. There are zero
integration tests — regressions are caught only by manual observation.

**One sentence:** "Stop babysitting the agent, finish the TUI feature set, prove it works with
tests."

*Source: scope-lock.md*

## 4. Functional Requirements

### 4.1 Reliability (G1)

| ID | Requirement | Priority | Acceptance Criteria |
|----|-------------|----------|---------------------|
| R1 | Port binding consistency | Must | Agent binds 7400 (gRPC) + 7401 (HTTP) on every start. Root cause of 8400 drift documented and fixed. |
| R2 | Memory optimization | Must | Steady-state < 250M with 5-10 sessions after 24h. Peak < 500M during burst registration. |
| R3 | Graceful shutdown | Must | On SIGTERM: (1) stop accepting new connections, (2) send GoingAway to all active gRPC streams, (3) wait up to 5s for drain, (4) exit 0. |
| R4 | TUI auto-reconnect | Must | When agent connection drops, TUI retries with exponential backoff (1s, 2s, 4s, 8s, max 30s). Status bar shows "reconnecting..." with attempt count. On reconnect, full session state refreshes. |
| R5 | DNS error reporting | Should | When gRPC connection fails due to DNS resolution, surface the hostname and resolution error in the TUI status bar. Do not silently retry. |

### 4.2 M3 Features (G2)

| ID | Requirement | Priority | Acceptance Criteria |
|----|-------------|----------|---------------------|
| F1 | Session Detail screen (T4) | Must | Full-screen detail view for a single session. Shows: session metadata (id, project, branch, cwd, type, started_at), status history (timeline of state transitions), agent info (which machine, connection status), and if streaming — live output. Navigate with `d` from dashboard. `q` returns. |
| F2 | Command Palette (T7) | Must | `Ctrl-P` opens overlay. Fuzzy-match search across: session names, project codes, agent names, and actions (attach, stop, start, health). Arrow keys navigate, Enter executes. Esc dismisses. Results update as user types. |
| F3 | Start Session from TUI (T13) | Must | `n` from dashboard opens "New Session" dialog. Select agent (from connected agents list), enter project code, enter cwd (with tab-completion from agent's project registry). Invokes StartSession RPC. New session appears in dashboard within 2s. |

### 4.3 Test Foundation (G3)

| ID | Requirement | Priority | Acceptance Criteria |
|----|-------------|----------|---------------------|
| T1 | gRPC integration tests | Must | Tests spin up agent gRPC server in-process (real tonic server, localhost binding). Test client connects and exercises: GetSessions, GetSession, RegisterSession, UnregisterSession, Heartbeat, StreamEvents, SendCommand, StopSession, GetHealth, ListAgents, ListProjects. Each RPC has at least one success-path and one error-path test. |
| T2 | TUI snapshot tests | Must | Use ratatui's TestBackend to render key screens and assert terminal output. Cover: dashboard (with sessions from 2 agents), stream view (with formatted messages), health screen, project list. Snapshots committed to repo for regression detection. |
| T3 | Graceful shutdown test | Must | Integration test: start agent, connect gRPC stream, send SIGTERM to agent process, verify stream receives GoingAway status before connection closes. |

### 4.4 Operational Maturity (G4)

| ID | Requirement | Priority | Acceptance Criteria |
|----|-------------|----------|---------------------|
| O1 | Config hot-reload | Should | Agent watches `~/.config/nexus/agents.toml` via inotify (Linux) / FSEvents (Mac). On change, re-read file and update internal agent list without restart. Log "config reloaded: N agents" at INFO level. |
| O2 | Binary size baseline | Should | Document release binary sizes in README. No optimization target yet — just measure and track. |

## 5. Technical Architecture

### 5.1 Stack (unchanged from MVP)

| Component | Technology |
|-----------|-----------|
| Agent | Rust, tonic 0.12, tokio, axum |
| TUI | Rust, ratatui, tonic, crossterm |
| IPC | gRPC (protobuf) + HTTP /health |
| Config | TOML (`~/.config/nexus/agents.toml`) |
| Build | Cargo workspace, GitHub Actions CI |

### 5.2 New Technical Components

| Component | Purpose | Crate |
|-----------|---------|-------|
| Shutdown coordinator | Tracks active streams, sends GoingAway, manages drain timer | nexus-agent |
| Reconnect manager | Exponential backoff reconnection with state refresh | nexus-tui |
| Command palette | Fuzzy matcher (nucleo or similar) + overlay widget | nexus-tui |
| Test harness | In-process agent server for integration tests | nexus-agent (tests/) |
| Snapshot test infra | TestBackend wrapper with assertion helpers | nexus-tui (tests/) |

### 5.3 Proto Changes

All proto changes must be **additive** (Nova is a live consumer). No field renumbering, no
removed RPCs, no changed semantics on existing fields.

New messages needed:

```protobuf
// Graceful shutdown notification (server → client on stream)
message GoingAway {
  string reason = 1;           // "agent shutting down"
  uint32 drain_timeout_ms = 2; // how long until force-close
}

// Session status history entry (for detail screen)
message StatusTransition {
  SessionStatus from = 1;
  SessionStatus to = 2;
  google.protobuf.Timestamp at = 3;
  string reason = 4;           // optional: why the transition happened
}
```

### 5.4 Memory Budget

| Component | Budget | Current (est.) | Notes |
|-----------|--------|----------------|-------|
| gRPC server overhead | < 20M | Unknown | tonic + hyper base allocation |
| Session registry | < 10M | Unknown | 50 sessions × metadata |
| Stream buffers | < 50M per stream | Unknown | Backpressure needed if unbounded |
| sysinfo collector | < 30M | Unknown | System metrics sampling |
| Protobuf codegen | < 10M | Unknown | Generated types |
| Total (5-10 sessions) | < 250M | ~450M | Profile to find the outlier |

**Investigation approach:** Use `heaptrack` or `jemalloc` profiling to identify the top 3
allocators. Likely culprits: unbounded stream buffers, Docker ps output caching, or protobuf
message cloning.

## 7. Scope & Constraints

### 7.1 In Scope

| Goal | Items |
|------|-------|
| G1: Reliability | Port binding fix, memory < 250M, graceful shutdown, auto-reconnect, DNS errors |
| G2: M3 Features | Session detail (T4), command palette (T7), start session (T13) |
| G3: Test Foundation | gRPC integration tests, TUI snapshot tests, shutdown test |
| G4: Ops Maturity | Config hot-reload, binary size baseline |

### 7.2 Out of Scope

- iMessage integration (I1-I5) — separate feature phase
- Prometheus /metrics endpoint
- Split pane stream views
- Web dashboard
- Memory target < 100M (< 250M is the target)
- Major architectural rewrites
- Hook migration stabilization (passive soak)

### 7.3 Hard Constraints

| Constraint | Detail |
|------------|--------|
| Proto backward compat | Additive changes only. No breaking changes to existing RPCs/messages. |
| Port binding | Must bind 7400 (gRPC) + 7401 (HTTP). No other ports. |
| Memory | < 250M steady-state, < 500M peak |
| Shutdown | Drain + GoingAway + 5s timeout |
| Platforms | Linux (systemd) + Mac (launchd), both working |
| Tests before features | G3 (test foundation) ships before or alongside G2 (features) |
