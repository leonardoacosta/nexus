4 architectural approaches to terminal session attach — trade-off analysis for Leo
Problem: CC sessions are bare processes on PTY devices — they do NOT run inside tmux. ssh + tmux attach does not work out of the box.
Topology: TUI on Mac → Tailscale → Agents on Linux homelab + Mac.
Goal: Let users jump into a running CC session from the TUI with full terminal interaction.
1

Nexus-Managed Sessions

Medium Scope
Nexus-agent spawns CC inside tmux: tmux new-session -d -s nx-<id> -- claude. Full attach works for nexus-started sessions. Ad-hoc sessions (Cursor, manual terminal) are stream-only. Creates a two-tier experience: managed sessions get full terminal, everything else gets read-only logs.

Pros

  • Full terminal for managed sessions — proven tmux pattern
  • Session lifecycle management (start, stop, restart) from TUI
  • tmux is battle-tested and cross-platform

Cons

  • Two-tier UX: managed sessions are first-class, ad-hoc are second-class
  • Scope creep: nexus becomes a session orchestrator, not just visibility+attach
  • Need new UI to start/manage sessions from TUI
  • Users must change workflow to get full attach
Key trade-off: Nexus was designed as a visibility and attach layer. This approach pulls it into session lifecycle management — a fundamentally different scope. The two-tier UX means most real-world sessions (Cursor, manual) remain stream-only.

Stream-Only (Drop Full Attach from MVP)

Low Scope
All sessions get gRPC stream attach only. Press 'a' to stream events as a rendered read-only log. No tmux, no terminal takeover, no session management. Full attach deferred to post-MVP.

Pros

  • Simplest MVP — least implementation effort
  • No scope creep: nexus stays as visibility layer
  • All sessions are treated equally (no two-tier UX)
  • No tmux dependency, no session lifecycle management
  • Can layer full attach on later without rework

Cons

  • No terminal interaction from TUI — read-only only
  • Manual SSH still required to intervene in a session
  • TUI is less useful without attach capability
  • Defers the hard problem, doesn't solve it
MVP calculus: If 80% of the value is visibility (seeing all sessions, their status, activity) and only 20% is attach (jumping in to interact), this approach ships the 80% first. Full attach can be added in a later iteration without reworking the core.
3

Hybrid — tmux Wrap on Launch Hook

High Risk
A CC SessionStart hook detects new sessions and attempts to wrap them in tmux retroactively. The fundamental problem: CC is already running when the hook fires — you cannot reparent a live process into a tmux session.

Pros

  • Would work for ALL sessions if it worked
  • No two-tier UX (uniform behavior)
  • Hook-based — no user workflow change

Cons

  • Cannot retroactively wrap a running process in tmux
  • Killing original + restarting loses CC context and session state
  • Race conditions between hook, original process, and new tmux session
  • Fragile: depends on CC hook timing and lifecycle
  • User confusion: session restarts mid-work
Fundamental blocker: Unix does not support reparenting a process's controlling terminal after it has started. The only workaround (kill + restart) destroys the CC session context, which defeats the purpose. This approach is architecturally broken.
4

PTY Relay via /proc

High Complexity
Agent captures the session's PTY output directly via /proc/<pid>/fd and relays it over a bidirectional gRPC stream. TUI renders full terminal output and sends keystrokes back. No tmux needed. Works for all sessions — but only on Linux.

Pros

  • Works for ALL sessions (no two-tier UX)
  • No tmux dependency
  • No hook changes or session management needed
  • No user workflow changes

Cons

  • Linux-only — /proc does not exist on macOS
  • PTY semantics are deeply complex (line discipline, signal handling, terminal modes)
  • Security: reading another process's fd requires same-user or root
  • Raw PTY output needs a terminal emulator in the TUI (vte crate)
  • Writing to stdin fd is unreliable for interactive programs
  • No macOS path without fundamentally different approach
Platform blocker: The TUI runs on Mac, and macOS agents have no /proc. This approach only works for Linux homelab agents. The TUI would need to handle two completely different attach codepaths per platform — or accept that Mac-hosted sessions can't be attached.

Decision Framework

Approach 3 (Hook Wrap) is architecturally broken — eliminate it. You cannot retroactively reparent a process's controlling terminal.

Approach 4 (PTY Relay) is technically interesting but platform-blocked. The TUI runs on Mac, macOS agents have no /proc, and PTY stdin injection is unreliable. Defer unless you want to invest in a terminal emulator (vte crate) and accept Linux-only full attach.

The real decision is between Approach 1 (Managed) and Approach 2 (Stream-Only):

• If nexus should be a session manager (start, stop, restart CC from TUI), go with Approach 1. Accept the two-tier UX and the increased scope. Full attach works for nexus-managed sessions; ad-hoc sessions are stream-only.

• If nexus should stay a visibility + monitoring layer, go with Approach 2 for MVP. Ship stream attach first, then layer full attach later once the right mechanism is clear. No rework required — stream attach is additive infrastructure regardless of which full-attach model you pick later.

Hybrid path: Start with Approach 2 (stream-only MVP), then add Approach 1 (managed sessions) as an opt-in feature post-MVP. Users who want full attach can start sessions via nexus; everyone else gets stream attach. This avoids scope creep in MVP while leaving the door open.
